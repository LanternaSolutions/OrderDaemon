# Internationalization (i18n) & Localization – Internal Guide

Audience: Core/Pro engineers, support, and doc writers

This page documents how Order Daemon loads and manages translations in core, with implementation notes tied to current code. Keep this page in sync with code changes.

Last reviewed: 2025-11-14

---

## Text Domain

- Domain: `order-daemon`
- POT template: `languages/order-daemon.pot`
- Compiled MO files: `languages/order-daemon-<locale>.mo`

## Load Timing and Strategy

Code reference: `src/Plugin.php::load_text_domain()`

1. Primary load (relative path)
   - Uses `load_plugin_textdomain('order-daemon', false, $rel_path)`
   - `$rel_path` is computed as `dirname(plugin_basename(ODCM_PLUGIN_FILE)) . '/languages'`
   - This is the standard WP mechanism that expects MO files under the plugin’s `/languages` folder.

2. Diagnostics
   - The method logs helpful diagnostics via `error_log()` while verifying setup:
     - `rel_path`, `plugin_basename`, `__FILE__`, `ODCM_PLUGIN_FILE`, (guarded) `WP_PLUGIN_DIR`
     - `locale`, `get_user_locale()` (if available)
     - Whether the domain is considered loaded by `is_textdomain_loaded('order-daemon')`

3. Fallback (absolute path)
   - If the domain isn’t loaded after the primary attempt, we try a direct MO load:
     - Resolve `$locale = determine_locale()` if available, otherwise `get_locale()`
     - Construct absolute MO path: `ODCM_PLUGIN_DIR . 'languages/order-daemon-' . $locale . '.mo'`
     - If the file is readable, call `load_textdomain('order-daemon', $mofile)`
   - This helps in environments where the relative-path mechanism fails (unusual plugin layouts, early init edge cases, etc.).

4. Final verification
   - Re-check `is_textdomain_loaded('order-daemon')`
   - Log summarized results of primary and fallback attempts

## Script Translations (JS)

If available, we register JSON translations for admin scripts using `wp_set_script_translations`:

- `order-daemon-admin-js`
- `order-daemon-rule-builder`
- `order-daemon-insight-dashboard`

Notes:
- Ensure these script handles match the ones used when enqueuing the assets.
- JSON files are typically generated by WordPress build tools (or GlotPress) and resolved by WP at runtime given the text domain.

## Adding/Updating Strings

- Use standard WP i18n wrappers in PHP, e.g. `__()`, `_x()`, `esc_html__()`, etc., with the domain `order-daemon`.
- For JavaScript, use `wp.i18n.__` and related helpers with the `order-daemon` domain.
- After adding strings:
  - Update the POT file (`languages/order-daemon.pot`) using your i18n extraction workflow (e.g., `wp i18n make-pot` or equivalent tooling).
  - Merge PO updates for target locales and compile to MO.

## Troubleshooting Checklist

- Verify MO file presence
  - `languages/order-daemon-<locale>.mo` exists and is readable by PHP.
- Confirm code path executes
  - `load_text_domain()` is hooked early enough and actually runs (check logs).
- Review logs
  - Look for the diagnostics emitted by `load_text_domain()` to confirm paths and outcomes.
- Double-check domain typos
  - All i18n calls must use `order-daemon` (exact match).

## Notes for Pro Extension

- Pro should not redefine the text domain; it reuses `order-daemon`.
- If Pro adds scripts that require translations, it should also call `wp_set_script_translations()` with the same domain for its handles.
- Avoid loading text domain twice; rely on core’s loader or guard re-loads appropriately.

## Change Management

- Any changes to script handles, asset pipeline, or plugin load order that affect i18n must be reflected here and in `docs/docs-plan.md` (A10 section).
- When altering the loader, keep absolute-path fallback and diagnostics unless there is a compelling reason to remove them, as they significantly aid support.
